34.5M 参数赢过 235B 大模型，三档全测完：PP-OCRv6 真强还是假强

作者：卡兹克 · 首发：2026-06-15 · 三档补测：2026-06-16 · 「数字生命卡兹克」

前段时间我写过一篇 OCR 选型实测，标题大概是，别瞎花钱了，我实测了 18 个文档、6 类场景，教你选对 OCR 模型。

那篇文章里，我当时的判断很简单，OCR 不是一个模型打天下。规整文本、手写、表格、双栏论文、公式文档，背后其实是几种完全不同的能力。你要的是逐字还原，还是文档理解；你能不能联网，数据能不能出本地；这些问题比"哪个模型分数最高"更重要。

结果那篇刚写完没多久，PaddleOCR 又丢了一个新东西出来，PP-OCRv6。

我一开始其实没太在意，OCR 更新嘛，正常，每年都有。直到我看到两行字，第一行是「Tiny 版本只有 1.5MB，可以直接跑在浏览器里」，第二行是「Medium 档只有 34.5M 参数，OCR 专项任务上超过了 Qwen3-VL-235B」。

我寻思了一下没寻思明白，34.5M 对 235B，6906 个汉字字符 vs 万亿 token 预训练，为什么专用小模型能在这件事上赢？

然后就是一个问题跟着另一个问题。OCR 的未来，真的是越来越大的 VLM 吗？还是说这条轻量专用模型的路，反而有自己的护城河？

这篇文章是上一篇的续集，也算是我对这个问题的亲手测验。我不是只跑了一个 Tiny demo，而是把 PP-OCRv6 Tiny、Small、Medium 三档都接进了本地 OCR Studio，沿用同一把 OmniDocBench 的尺子，加了四张真实世界难图，逐档切换模型实测了一遍。说说我看到的结果，以及我自己对这件事的判断。

· · ·

稍微说一下背景，免得有朋友没接触过这个领域。

OCR 就是"把图片里的字变成可编辑文本"这件事。PaddleOCR 是百度开源的 OCR 系统，在国内工程圈用得很广。PP-OCRv6 是这套系统的最新版，六月十一号刚发的。

现在主流的 OCR 有两条路，一条是传统的检测+识别流水线，先框出文字区域，再逐框读出文字，轻量，可以在端侧跑。另一条是用多模态大模型直接看图说字，像 GPT-4o、Qwen3-VL 那样，精度高但很重，必须调云端 API。PP-OCRv6 走的是第一条路，但这次我想亲手戳一戳的是，它到底怎么用这么小的模型，去和一堆大模型抢 OCR 这个活。

· · ·

不过在说结果之前，先说说这次跑分是怎么做的。

六个选手，18 张文档图片，逐张手动对比不是人干的事。所以我先搭了个本地 OCR 工作台，FastAPI + ONNX Runtime 做后端，Apple Silicon 自动走 CoreML 加速，Tiny、Small、Medium 三档模型都能在设置页里一键切换，然后用脚本把所有图片一次性打过去。

拖拽上传区，支持批量处理和剪贴板粘贴（⌘V 截图直接粘）。
报纸页识别结果，193 个文字框，右侧是全文转录。
历史记录，39 条，每张带缩略图、框数和耗时。
三档模型一键切换，CoreML 开关，检测参数可调。

识别完可以看框选可视化，导出 CSV/Excel，历史记录全留着。OmniDocBench 的 18 张图就是用这台工作台逐一跑完的，Apple Vision 那边用 ocrmac 调系统 API 跑同一套图做对照。

这次跑了六个选手，本地三档，云端两个参照，

PP-OCRv6 Tiny，1.5MB，onnxruntime-web 跑在 Chrome 浏览器里，WebGPU 加速，图片全程不出本地。

PP-OCRv6 Small，7.7MB，本地 ONNX Runtime + CoreML，Apple Silicon 加速。

PP-OCRv6 Medium，34.5MB，本地 ONNX Runtime + CoreML，三档里精度最高。

Apple Vision，macOS 原生 OCR，accurate 档，完全离线，本地基线。

GLM-OCR，智谱云端 OCR API，参照系。

PaddleOCR-VL-1.5，百度多模态大模型云端 API，参照系。
· · ·

先说三档模型。PP-OCRv6 的三档共用同一套 LCNetV4 骨干，检测和识别都复用这套结构，区别只在宽度和深度。Tiny 跑在浏览器里，Small 适合移动端或本地轻量应用，Medium 是精度优先的本地部署档。

官方在 OCR 专项任务上的数据是这样的，检测/识别分（分越高越好），Medium 86.2/83.2，Small 84.1/81.3，Tiny 80.6/73.5。对比一下 Qwen3-VL-235B 的 38.3/74.9，这就是发布会说的那个"34.5M 赢 235B"的来源。

不过要理解为什么专用小模型能赢，有一个关键点，PP-OCRv6 的架构专门针对 OCR 这件事做了优化，检测侧感受野从 3×3 扩到 7×7 对小字和密集文字更友好，识别侧加了轻量注意力模块处理字符上下文，字典覆盖中英日和 46 种拉丁语系共 50 种语言。模型不大，但任务非常对口。

检测和识别两个 OCR 专项任务上，PP-OCRv6 三档都拉开了和 VLM 的差距。
Tiny / Small / Medium 三档覆盖浏览器、移动端、本地应用和服务器部署。

Apple Vision 那份结果，这还是 Apple 的亲儿子框架、accurate 档。手写中文确实难，但 PP-OCRv6 Tiny 的编辑距离是 0.283，Medium 是 0.228，Apple Vision 是 0.570，差了整整一倍多。而且三档之间的梯度很清晰，手写场景是最能体现参数规模价值的场景，这条路值得继续走。

· · ·

当然，不是所有场景 PP-OCRv6 都占优，必须说失分的地方。

最大的坑是双栏排版。中文学术书籍、英文期刊论文，这类图片 PP-OCRv6 和 Apple Vision 都垮得很惨，编辑距离高达 0.8 到 0.9。

测试样本，中文电子书，典型双栏排版

原因很直接，检测+识别的流水线，检测阶段只知道"这里有一块文字，坐标是 xxx"，它不理解"左边这一列要先读完，再读右边那一列"这种版式逻辑。识别结果按文字框从上到下的空间顺序排列，把左右两列混编在一起了。

这类场景是 GLM-OCR 和 PaddleOCR-VL 的主场。多模态模型能理解文档版式，能正确处理双栏、表格、公式，pp-ocrv6 这种轻量流水线在版式复杂的文档上确实搞不定。这不是 Medium 比 Tiny 大就能解决的，是结构性限制。

· · ·

除了 OmniDocBench 的标准文档，我还额外测了四张真实世界的难图，覆盖透视变形、点阵字体、浮雕低对比度文字和七段数码管四个场景。

五个选手，四张图，按场景说。

名片透视，斜拍加彩色底，透视角度让字体变形是最大的难点。Medium 最完整，完整读出「DESIGN SOUL / JONATHAN DOE / Creative Designer」加电话和网址；Small 漏掉了 Creative Designer 这行；Tiny 只读出名字和品牌。Apple Vision 在名字识别上有错字。

点阵字体，字形断裂对字符集要求高，反而是 Small 最稳，两行都完整识别，DotMatrix 和 RETRO PRINT CHARM 都没问题。Medium 同样稳，Tiny 识别 DotMatrix 字体时偶有字符缺失。

轮胎压印，低对比浮雕字是边界场景的代表，肉眼都不容易看清的那种。Medium 读出 TREADWEAR 220 和 PLACARD IN VEHICLE；Apple Vision 只读出了「220」，其余全丢；Tiny 读出了 TREADWEAR 但后续行识别不稳。

电梯数码屏，七段数码字体加反光金属底，对所有模型都有挑战。三档 PP-OCRv6 都识别出了产品编号 BVY413HSW、BVY411HSW、品牌名 ORB ELEVATOR 和网址 orbelevator.en.alibaba.com，Apple Vision 在反光区域有漏检。

速度参考，Tiny 大约 3–15 秒一张，Small 4–25 秒，Medium 10–52 秒，Apple Silicon CoreML 加速，Apple Vision 0.16–0.54 秒，PaddleOCR-VL 走云端大约 6.7–15.9 秒。

· · ·

这里有件事我想多说几句，它是整个「为什么 34.5M 能赢 235B」问题的核心。

大模型做 OCR，有个很有趣的缺陷，幻觉。不是那种「胡说八道」的幻觉，而是更隐蔽的，它会把图片里写错的字「纠正」过来。手写笔记里的错别字，VLM 看到后，不是忠实地抄下来，而是读成正确的写法。从「理解语义」的角度看是聪明的，但从「逐字还原」的角度看，这是一个 bug，不是 feature。

公开数据里有一个很有意思的指标，PP-OCRv6 精确匹配率 93.2%，Qwen3-VL-235B 是 80.6%。差了将近 13 个百分点。这个指标衡量的是「识别出的文字是否和图片完全一致」，不允许任何「聪明的修正」。

所以如果你需要的是发票、合同、表单、票据这类逐字准确性要求极高的场景，大模型的「补全语义」能力反而会变成风险。专用轻量模型，读到什么就是什么，没有脑补，更可靠。这是我认为 PP-OCRv6 能在 OCR 专项任务上赢过大模型的根本原因，任务范式不一样，OCR 要的是抄写员，不是翻译家。

· · ·

三档跑完，我对 PP-OCRv6 的判断反而比一开始更清楚了。

Tiny 适合浏览器或嵌入式，1.5MB，图片全程不出本地。Small 是甜点档，7.7MB，速度和精度的均衡点，手写提升可以感知。Medium 是本地精度优先的选择，手写、彩色教材这类边界场景提升明显，适合对精度有要求但不想调 API 的本地部署。

三档共通的是，数据可以不出本地，不需要 API key。差别在部署形态：Tiny 可以做成纯前端在线版，打开网页就跑；Small 和 Medium 我这次是放在 FastAPI + ONNX Runtime 的本地 Studio 里跑，前端负责上传、历史记录、框选可视化和表格导出，后端负责加载模型推理。对很多场景来说这是刚需，敏感文档的公司内网，离线环境，不想为每次 OCR 付 API 费用的个人开发者。

OCR 的未来是不是越来越大的 VLM？我的判断是，两条路各有各的场景。大模型擅长「理解文档」，能处理双栏、公式、表格结构；轻量专用模型擅长「抄写文字」，逐字准确，无幻觉，能在端侧跑。当你需要的是逐字还原，专用模型反而更靠谱。所以别瞎花钱，看清楚自己的场景，再选工具。
